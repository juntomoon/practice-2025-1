# Техническое описание разработки Telegram‑бота «Погодный ассистент»

## Назначение проекта
Бот предоставляет пользователям актуальный прогноз погоды с персонализированными рекомендациями по одежде. Ключевые особенности:
*   **Персонализация**: учёт пола пользователя и типа активности (прогулка, бег, поход).
*   **Контекстность**: советы адаптируются под ощущаемую температуру, ветер, осадки и влажность.
*   **Диапазон температур**: вместо одной цифры бот показывает суточный минимум и максимум.
*   **Автоматизация**: ежедневная рассылка прогноза по избранным городам.
*   **Часовой пояс**: рассылка срабатывает по локальному времени домашнего города пользователя, а не сервера.

## Ключевые модули и их функционал

### 1. Модуль конфигурации и инициализации
*   **Загрузка переменных окружения**: Считывает `BOT_TOKEN` и `OPENWEATHER_API_KEY` из файла `.env`. Это предотвращает утечку секретных ключей при публикации кода.
*   **Инициализация бота**: Создание экземпляра `telebot.TeleBot`.
*   **Структура данных**: Определение пути к файлу базы данных (`users_prefs.json`).

### 2. Модуль работы с базой данных (JSON)
Так как проект не требует высокой нагрузки или сложных связей, используется легковесное файловое хранилище.
*   `load_users()`: Читает JSON-файл. Если файла нет, возвращает пустой словарь. Обрабатывает ошибки чтения (поврежденный файл).
*   `save_users(data)`: Сериализует словарь в JSON с кодировкой UTF-8 и отступами для читаемости.
*   **Структура записи пользователя**:
    ```json
    {
      "user_id": {
        "home_city": "Москва",
        "home_timezone_offset": 10800,       // Смещение в секундах от UTC
        "daily_time": "08:00",              // Время рассылки
        "favorites": ["Казань", "Сочи"],   // Список избранных городов
        "gender": "female",                // Пол для советов
        "activity": "обычная"              // Последняя выбранная активность
      }
    }
    ```

### 3. Модуль получения погодных данных
Реализован через два основных метода API OpenWeatherMap:
*   **`get_weather(city)`**:
    *   Запрос текущей погоды (`/data/2.5/weather`).
    *   Возвращает температуру, влажность, скорость ветра, описание и **смещение часового пояса** (`timezone` в секундах).
    *   Нормализует данные об осадках (дождь/снег).
*   **`get_daily_temp_range(city)`**:
    *   Запрос прогноза на 5 дней (`/data/2.5/forecast`).
    *   Анализирует первые 8 записей (интервал 3 часа = 24 часа).
    *   Вычисляет минимальную и максимальную температуру за сутки.
    *   Это решает задачу отображения диапазона температур, а не только текущего значения.

### 4. Модуль генерации советов по одежде (`get_clothing_advice`)
Логика построена на каскадных условиях (`if/elif`), учитывающих:
1.  **Температуру по ощущениям** (`feels_like`) — это важнее сухой цифры термометра.
2.  **Пол пользователя**: Формулировки советов различаются (например, упоминаются леггинсы/платья для женщин, специфические слои одежды для мужчин).
3.  **Тип активности**:
    *   *Прогулка*: Стандартный совет.
    *   *Бег/Велосипед*: Акцент на дышащие материалы и защиту от ветра.
    *   *Поход*: Акцент на многослойность и термобельё.
4.  **Погодные факторы**: Ветер усиливает охлаждение, дождь требует непромокаемой одежды.

### 5. Машина состояний (FSM)
Для диалогового взаимодействия используется простой словарь `user_states`, где ключом является ID пользователя, а значением — текущее состояние.
*   **Состояния**: `'set_home_city'`, `'add_favorite_city'`, `'choose_activity'`, `'ask_city_with_choice'`, `'ask_city_direct'`, `'set_daily_time'`.
*   **Механизм**: Хендлеры проверяют `user_states.get(uid).get('state')`.
*   **Критическая логика**: При любом завершении диалога (успех или ошибка) состояние **обязательно** удаляется (`pop`), чтобы избежать зависания бота в режиме ожидания ввода.
*   **Защита от зацикливания**: Во всех состояниях ввода текста добавлена явная проверка на команду «🏠 Главное меню». Если пользователь пишет её, состояние сбрасывается мгновенно, минуя логику поиска города.

### 6. Система меню
Используется `ReplyKeyboardMarkup` для навигации.
*   **Главное меню**: Стартовая точка.
*   **Меню избранного**: Динамически генерирует кнопки удаления для каждого сохраненного города.
*   **Диалоговые меню**: Показывают контекстные кнопки (например, «Использовать домашний город» или «Ввести другой»).

### 7. Планировщик рассылок (Scheduler)
Реализован как отдельный поток (`threading.Thread`), работающий в фоне.
*   **Проблема старого подхода**: Сравнение времени сервера с настроенным временем.
*   **Решение**:
    1.  Берется текущее время в UTC.
    2.  Для каждого пользователя берется сохраненное смещение часового пояса (`home_timezone_offset`).
    3.  Время конвертируется в локальное время пользователя.
    4.  Сравнивается с сохраненным временем рассылки.
*   **Нюанс летнего времени**: OpenWeatherMap возвращает смещение, актуальное на момент запроса. При смене сезона смещение изменится, и рассылка автоматически подстроится под новое местное время.

## Алгоритм работы основных сценариев

### Сценарий 1: Получение прогноза «здесь и сейчас»
1.  Пользователь нажимает **«🌤 Погода и одежда»**.
2.  Бот спрашивает про **активность** (кнопки).
3.  Бот предлагает выбрать **город** (использовать домашний или ввести новый).
4.  Если город введен вручную или выбран:
    *   Бот отправляет статус «⏳ Проверяю...».
    *   Делает запрос к API погоды.
    *   Делает запрос на диапазон температур.
    *   Генерирует текст с учетом пола и активности.
    *   **Важно**: Очищает состояние пользователя и показывает главное меню.

### Сценарий 2: Настройка времени рассылки
1.  Пользователь должен сначала задать **домашний город** (иначе кнопка времени блокируется).
2.  При задании города API возвращает смещение часового пояса, которое сохраняется в БД.
3.  Пользователь вводит время в формате `ЧЧ:ММ`.
4.  В фоне планировщик конвертирует UTC время сервера в локальное время этого города и отправляет рассылку.

### Сценарий 3: Работа с избранным
1.  Пользователь добавляет город.
2.  Система проверяет существование города через API.
3.  Город добавляется в массив `favorites` в JSON.
4.  При ежедневной рассылке бот проходит циклом по всем городам из `favorites` этого пользователя.

### Сценарий 4: Обработка ошибок и выход
*   Если пользователь в режиме ввода города пишет **«🏠 Главное меню»**:
    *   Срабатывает проверка внутри хендлера состояния.
    *   Состояние удаляется из памяти.
    *   Вызывается функция `show_main_menu`.
    *   Цикл ввода прерывается корректно.

## Особенности реализации и решения проблем

| Проблема | Решение |
| :--- | :--- |
| **Бот не выходит в главное меню после ошибки** | Сброс состояния (`user_states.pop`) перенесен в начало функции обработки запроса, до любых проверок на успех/ошибку. |
| **Рассылка идет по времени сервера** | Внедрена конвертация времени через смещение (`timezone offset`), полученное от API погоды при сохранении домашнего города. |
| **Отсутствие диапазона температур** | Реализован отдельный запрос к методу `forecast`, который парсит 8 ближайших записей и вычисляет min/max. |
| **Лишний текст при отсутствии данных** | Логика формирования сообщения явно проверяет наличие `min_temp` и `max_temp`. Если их нет, блок с диапазоном просто не добавляется . |
| **Ввод некорректного времени** | Строгая валидация формата `ЧЧ:ММ` и диапазонов чисел (0-23, 0-59) с понятными сообщениями об ошибке. |

## Инструкция по запуску

1.  **Подготовка окружения**:
    ```bash
    python -m venv venv
    source venv/bin/activate  # Windows: venv\Scripts\activate
    pip install pyTelegramBotAPI requests python-dotenv
    ```
2.  **Настройка секретов**:
    Создайте файл `.env` в корне проекта:
    ```env
    BOT_TOKEN=ваш_токен_от_botfather
    OPENWEATHER_API_KEY=ваш_ключ_от_openweathermap
    ```
3.  **Запуск**:
    Запустите скрипт `python main.py` (или как он у вас назван).
    При первом запуске автоматически создастся файл `users_prefs.json`, если его нет.

## План развития (Roadmap)

1.  **Кэширование погоды**: Сохранение прогноза на 1 час, чтобы не дергать API при каждом запросе одного и того же города.
2.  **Расширенные настройки**: Добавление предпочтений по стилю одежды (спортивный, деловой) или чувствительности к холоду.
3.  **Уведомления о шторме**: Отправка экстренных сообщений при резком ухудшении погоды (гроза, ураган) сверх ежедневной рассылки.
4.  **Графики**: Отправка изображений с графиками изменения температуры в течение дня.
